Double Dereferencing:
You can make variables with the value of another variable, and dereference it up to two times.
var(name = "apple")
Here, a variable is being created, called "name", with a value of "apple". Now, we can use the variable's value to make a new variable. Let's try it out:
var(@name = 50)
A variable is now made with the value of the variable created previously ("name"). So, a new variable called "apple" was just made with a value of 50.
paste(text"@name")
Output: "apple"
Dereferencing once turns "@name" into "apple".
paste(text"@@name")
Output: 50
Now, because we're dereferencing twice, "@@name" dereferences into "@apple". On the second dereference, "@apple" turns into 50. This mechanic is useful for making new variables on-the-fly, like when you're coding programming language in Direct+.
Direct+ supports only two dereference levels, for both speed and safety. Chances are, you probably won't need any more than two levels. If you accidentally use more than two levels, a Variable Reference error will occur.
Full Program:
var(name = "apple")
var(@name = 50)
paste(text"@name")
paste(text"@@name")
Output: "apple"
Output: 50
Implicit Loop Counters
Sometimes, Direct+'s implicit loop counter, which is great for simple tasks, isn't enough for custom iteration or non-linear updates. This is why you can simply omit it from the loop and use a custom counter instead.
Example (Implicit Counter):
loop(5, i) <<
  paste(text"@loop:i")
>>
Output: 1
Output: 2
Output: 3
Output: 4
Output: 5
Example (Explicit Counter):
var(i = 0)
loop(5) <<
  add(@i, 1 => i)
  paste(text"@i")
>>
Output: 1
Output: 2
Output: 3
Output: 4
Output: 5
You'll notice that the output is the same, and that's because we set up the explicit counter so it works the same way as the implicit one. Try incrementing the counter by 2, decrementing, or starting at a different number, and watch the output change. You can see that this is way more flexible than the implicit version.
Parameterized Functions, Namespaces, and Return Values:
Most times, when writing a function, you want to include parameters inside of that function. Those parameters can be used to provide additional functionality to that function, as well as make your code cleaner, so you don't have to rely on global variables all the time. The namespaced variables also get deleted upon function end, which is great to save program memory.
To make a function with parameters, you use the standard label syntax, followed by straight brackets and your variable names listed in them, as shown:
/label_name[some, parameters, here]
In the example above, three variables are being defined: "some", "parameters", and "here". Now, how do we access their values? Well, that's where namespacing comes in. Since they're part of a function, then they must be prefixed with their parent function's name, as shown:
@label_name:some
@label_name:parameters
@label_name:here
General Example: @function:variable
Now that we can get and access variables, it's time to make a result out of them. Extending upon the /return label, you can add a value into it using straight brackets, as shown:
/return[something]
Note that only one return value at a time is supported. (Lists can also be returned, just in the form of a variable) Execution of the function immediately stops as soon as /return is found.
Example (Unparameterized Function):
var(text = "hello, world!")
run(/print)
;
/print
  paste(text"@text")
/return
;
Output: "hello, world!"
Example (Parameterized Function):
run(/print["hello, world!"])
;
/print[text]
  paste(text"@print:text")
/return
;
Output: "hello, world!"
You can see that while the unparameterized version is more clear and explicit, it does have some issues that are fixed by the parameterized version. For example, the fact that the variables are not namespaced is less safe since variable names may conflict, and they're not auto-deleted.
Multiple Files, Variable Sharing, and Cross-file Functions:
Files in Direct+ share a runtime environment after being imported, but keep logical separation through namespaces and import rules.
To import a file, you run the use() command at the top of your main code file. Don't use any file extensions when importing, just use the base name of the file, as shown:
main
use(filename here)
Important things to note:
1. Imports can only be nested one level deep. For example, if [main] imports [utils], [utils] can't import any other files.
2. Imported variables are automatically namespaced by file. For example, if [utils] defines "x", then it's accessed as @utils:x after importing.
3. Each file has its own namespace. Variables with the same name in different files do not conflict.
4. Imported variables can't be changed from outside their parent file. If you attempt to change their values from a different file, a Namespacing Collision error will occur.
5. Think of imports as extended functionality to your main program. They can't directly modify the state of your main program, and you can't directly change anything they do on a code level. Modules just provide more tools for you to use in your programs.
6. Importing just adds variables and defines which functions can be accessed from which file. Once again, imports just provide additional functionality to your main file.
7. No file extensions are added when naming a file to import. Just use the base name of a file.
To call a function from a different file, use a label prefixed with its parent file, as shown:
main
run(/file:function_name)
Parameters, return values, and arrow assigment all work normally.
General Example:
main
use(utils)
var(result)
run(/utils:mul_by_2[5] => result)
paste(text"@result")
;
utils
var(result)
;
/mul_by_2[num]
  mul(@mul_by_2:num, 2 => result)
/return[@result]
;
Output: 10
List Math:
Direct+ supports math operations on lists through the built in add/sub/mul/div/mod/sqrt/random commands, which also work with numbers.
There are four cases to remember when using list math:
1. If both inputs are numbers, the output is a number.
2. If both inputs are lists, the output is a list. Each item in the output list is the result of the operation being performed on the corresponding items in the input lists. If the lists are different lengths, then a Lists are Different Lengths error will occur.
3. If one input is a list and the other is a number, the output is a list. The input number is applied to each item in the list.
4. If there's just one input parameter and the input is a list, then the output is a list. The input list is used as the singular input.
Examples:
var(num1 = 5)
var(num2 = 10)
var(list1 = [5, 10, 15])
var(list2 = [2, 4, 6])
add(@num1, @num2 => result) $ 5 + 10 = 15
add(@list1, @list2 => result) $ [5, 10, 15] + [2, 4, 6] = [7, 14, 21]
add(@list1, @num1 => result) $ [5, 10, 15] + 5 = [10, 15, 20] (Alternative parameter order is supported)
sqrt(@list1 => result) $ [5, 10, 15] = [2.236, 3.162, 3.873]
Building Blocks
Here are just some Direct+ Primitives that can be easily added to your programs so you don't have to think much about how to make a while loop or linked list.
While Loop:
loop(0) << $ start with an infinite loop for polling
  if(some condition) << $ if some condition is met, break the loop
    break()
    >>
  $ add code to do something here (this is the while loop body)
  >>
Linked List:
Required Setup:
1. var(temp) $ a temporary variable to hold pointers in /get_pointer
2. var(pointer) $ a variable to hold the result of /get_pointer
3. var(current_node = 0) $ the current node in the linked list
4. /get_pointer[node] $ returns the pointer to the next node
     item(@get_pointer:node, 2 => temp)
     /return[@temp]
define variables at the top of your program
add the /get_pointer function to the end of your main program after ;
$ define linked list
var(head = "node1") $ pointer to the start of the linked list
var(node1 = {"apple", "node2"}) $ first node with a value of "apple" points to node2
var(node2 = {"banana", "node3"}) $ second node with a value of "banana" points to node3
var(node3 = {"cherry", "node4"}) $ third node with a value of "cherry" points to node4
etc.
$ traverse linked list
var(current_node = @head) $ sets the current node to @head ("node1")
item(@@current_node, 1 => dest) $ gets value of the current node
run(/get_pointer[@@current_node] => pointer) $ gets the pointer to the next node
var(current_node = @pointer) $ sets the next node to the pointer obtained from /get_pointer
etc.
Linked list "shapes":
1. Linear linked list: each node points to the next node, and the last node points to null.
2. Circular linked list: each node points to the next node, and the last node points to the first node.
3. Doubly linked list: each node points to both the previous and next nodes.
The /get_pointer function can be modified to support doubly linked lists and other structures.
